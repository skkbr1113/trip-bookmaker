# skills/warikan.md

## 目的
旅行メンバーが費用をリアルタイムで入力・精算できる割り勘ページ（warikan.html）を作ること。
brand.mdとproject.mdの内容を踏まえて実装すること。

## 変数（毎回質問すること）
割り勘ページ生成前に以下を必ず確認すること。

- メンバーの名前（人数分。人数は問わない）
- Firebase接続情報（apiKey・authDomain・databaseURL・projectId）
- スマホ特化（max-width 420px）にするか、PC対応も含めるか

## 機能仕様（F1〜F5）

### F1：支払い入力フォーム
- 支払者：メンバーから1人選択（タップしやすいボタンまたはセレクト）にすること。
- 用途：テキスト入力にすること。
- 金額：数値入力・整数・円にすること。inputmode="numeric"でスマホのテンキーを出すこと。
- 割り勘対象者：メンバー全員のチェックボックス。初期状態は全員チェック済みにすること。
- 登録ボタン：押すとFirebaseに追加し、フォームをクリアすること。

### F2：支払い一覧
- 登録済みデータを一覧表示すること（支払者・用途・金額・対象者が分かる形）。
- 各行に編集ボタンと削除ボタンを設けること。
- 編集：その行の内容をフォームに戻して上書き保存できること。
- 削除：確認ダイアログを挟んでから削除すること。
- データが0件のときは空状態の文言を表示すること。

### F3：精算計算（最小送金回数）
- 各支払いの分担：amount ÷ participants人数（floor除算）。余りは支払者負担にすること。
- 各メンバーの収支：支払った合計 − 負担すべき合計。
- 最小送金回数アルゴリズム：収支のプラス・マイナスを絶対値の大きい者どうしから貪欲に相殺すること。
- 表示例：「おゆ → ゆーと ¥2,500」の形式で表示すること。
- 全員精算済みの場合は「精算の必要はありません」と表示すること。

### F4：サマリー
- 合計金額（全支払いの総額）を表示すること。

### F5：リアルタイム同期
- Firebase Realtime DatabaseのSDK（compat版・CDN）を使うこと。
- db.ref('payments').on('value', ...) で全員の画面にリアルタイム反映すること。

## F6について
レシート読取機能（F6）は実装しないこと。対象外として扱うこと。
ただし後から追加しやすいよう、入力フォームの金額・用途欄への値セットを1か所にまとめた関数構成にしておくこと。

## Firebaseデータモデル（/payments/{自動ID}）
```
{
  payer:        "メンバー名",
  purpose:      "用途（自由入力）",
  amount:       5000,
  participants: ["メンバー名", ...],
  createdAt:    timestamp
}
```

- 追加：db.ref('payments').push({...})
- 監視：db.ref('payments').on('value', snap => 再描画(snap.val()))
- 編集：db.ref('payments/' + id).update({...})
- 削除：db.ref('payments/' + id).remove()
- 一覧はcreatedAt昇順で表示すること。

## セキュリティ注意事項
- Firebase apiKeyはウェブ用の公開前提のキーなのでソースに直書きしてよいこと。
- Realtime Databaseはテストモード（誰でも読み書き可能）で作成する前提。
- 旅行後にルールを締めるか、データを削除する旨をコードのコメントに残すこと。

## ナビゲーションバーの仕様
ページ下部に固定ナビを設置すること。

```html
<nav class="bottom-nav">
  <a href="index.html" class="nav-item">📋 しおり</a>
  <a href="warikan.html" class="nav-item active">💰 割り勘</a>
</nav>
```

## 技術仕様
- 単一HTMLファイル（HTML・CSS・JS内包）にすること。
- デザインはbrand.mdに従い、index.htmlが存在する場合は必ず参照して揃えること。
- スマホ特化かどうかは変数確認の結果に従うこと。

## 完成条件（受け入れ基準）
実装後に以下を確認し、不具合があれば修正すること。

1. しおり（index.html）と見た目の統一感があること。
2. 支払いの入力・編集・削除が動くこと。
3. 複数端末で開いた状態で、片方の入力がもう片方にリアルタイム反映されること。
4. 精算結果・合計・一人当たりが正しく計算・表示されること。
5. 割り勘対象者を一部だけにした支払いも精算に正しく反映されること。
6. 下部ナビでしおり↔割り勘を行き来できること。
