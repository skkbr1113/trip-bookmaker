# 軽井沢旅行サイト ― 割り勘ページ実装依頼（Claude Code用）

軽井沢旅行サイトの「割り勘ページ」を作ってください。スマホブラウザ特化・1ファイル構成・GitHub Pages公開を前提とします。今回は **F1〜F5 を実装**します（レシート自動入力＝F6 は今回対象外。詳細は後述）。

---

## 0. 前提・環境

- リポジトリ：`skkbr1113/karuizawa-trip`（GitHub Pages で公開予定）
- 既に存在するファイル：`index.html` … 旅行しおりページ（静的）
- 今回作るファイル：**`warikan.html`**（リポジトリ直下、単一ファイルに HTML / CSS / JS を内包）
- データ保存：**Firebase Realtime Database**（無料 Spark プラン、テストモードで作成済み）
- メンバーは5人固定でハードコード：**なっちゃん / あゆ / おゆ / のぶ / ゆーと**
- 通貨は日本円のみ・整数（小数なし）

**最初に必ず `index.html` を開いて、カラー・フォント・角丸・破線枠などのスタイルを読み取り、割り勘ページもそれに揃えてください。** 2ページで世界観を統一したいです。

（apiKey はウェブ用の公開前提のキーなので、ソースに直書きで問題ありません。）

```javascript

  authDomain: "karuizawa-trip.firebaseapp.com",

  projectId: "karuizawa-trip",
};

const db = firebase.database();
```

### データモデル（`/payments/{自動ID}`）

```
{
  payer:        "なっちゃん",                       // 支払った人
  purpose:      "ガソリン代",                        // 用途（自由入力）
  amount:       5000,                                // 金額（整数・円）
  participants: ["なっちゃん","あゆ","のぶ"],         // この支払いを割り勘する対象者
  createdAt:    1700000000000                        // 並び順用（ServerValue.TIMESTAMP か Date.now()）
}
```

### 同期方針

- 追加：`db.ref('payments').push({...})`
- 監視：`db.ref('payments').on('value', snap => 再描画(snap.val()))` … **全員の画面にリアルタイム反映**されること
- 編集：`db.ref('payments/' + id).update({...})`
- 削除：`db.ref('payments/' + id).remove()`
- 一覧は `createdAt` 昇順で表示

## 3. 機能仕様（F1〜F5）

### F1：支払い入力フォーム
- **支払者**：5人から1人選択（タップしやすいボタン or セレクト）
- **用途**：テキスト入力
- **金額**：数値入力（整数・円）
- **割り勘対象者**：5人のチェックボックス。**初期状態は全員チェック済み**。1人だけ・一部だけの割り勘も可能に
- **登録ボタン**：押すと Firebase に追加し、フォームをクリア

### F2：支払い一覧
- 登録済みデータを一覧表示（支払者・用途・金額・対象者がわかる形）
- 各行に **編集** と **削除** ボタン
- 編集：その行の内容をフォームに戻して上書き保存できる（モーダル or インライン、実装しやすい方で可）
- 削除：**確認ダイアログ**を挟んでから削除
- データが0件のときは空状態の文言を表示

### F3：精算計算（最小送金回数）
「誰が誰にいくら払えば精算できるか」を算出して表示します。

- **各支払いの分担**：その支払いの `amount` を `participants` の人数で等分。
  - 1人あたり負担 = `floor(amount / 人数)`
  - 割り切れない余り（円）は **支払者(payer) が負担**（payer の負担に余りを上乗せ）
- **各メンバーの収支** = （その人が支払った合計）−（その人が負担すべき合計）
  - プラス＝受け取る側 / マイナス＝払う側
- **最小送金回数アルゴリズム**：収支がプラスの人とマイナスの人を、絶対値の大きい者どうしから貪欲に相殺していき、送金の本数を最小化する
- 表示例：`おゆ → ゆーと ¥2,500` のような行を必要数だけ並べる
- 全員が清算済み（収支ほぼ0）なら「精算の必要はありません」等を表示

### F4：サマリー
- **合計金額**（全支払いの総額）
- **一人当たり金額**（合計 ÷ 5）

### F5：リアルタイム同期
- F1〜F4 すべてが Firebase 経由でリアルタイムに反映されること（上記「同期方針」参照）

### 下部ナビゲーション（2ページ接続）
ページ下部に固定ナビを置き、しおり↔割り勘 を行き来できるように。
```html
<nav class="bottom-nav">
  <a href="index.html" class="nav-item">📋 しおり</a>
  <a href="warikan.html" class="nav-item active">💰 割り勘</a>
</nav>
```
※ `index.html`（しおり）側にも同じナビを後で追加する想定です。今回は warikan 側に実装し、index 側は別途対応とします（やれそうなら index.html にも同じナビを足してもらってOK）。

## 4. 今回やらないこと（F6・将来の後付け）

- **F6（レシート写真の自動読み取り）は今回実装しません。** Anthropic API キーは渡しません。
- ただし**後から足しやすい作り**にしてください。具体的には、F1の入力フォーム付近に「将来ここにレシート読取ボタンを置く」程度の余地を残し、金額・用途をプログラムからフォームに流し込める関数構成（入力欄への値セットを1か所にまとめる等）にしておけば十分です。**過剰な作り込みは不要**です。

## 5. セキュリティ・注意（コードにコメントで明記）

- Realtime Database は現在**テストモード**＝誰でも読み書き可能な状態です。短期の身内利用前提なので今はこのまま進めますが、**旅行後にルールを締める or データを削除する**旨をコードのコメントに残してください。

## 6. 完成条件（受け入れ基準）

1. スマホで `warikan.html` を開くと、しおり（index.html）と統一感のある見た目で表示される
2. 支払いの**入力・編集・削除**が動く
3. 複数端末で開いた状態で、誰かが入力すると**全員の画面に即反映**される
4. **精算結果（誰が誰にいくら）・合計・一人当たり**が正しく計算・表示される
5. 割り勘対象者を一部だけにした支払いも、精算に正しく反映される
6. 下部ナビで しおり↔割り勘 を行き来できる

実装後、上記1〜6を実際に動かして確認し、不具合があれば修正まで行ってください。
