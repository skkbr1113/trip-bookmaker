# 旅行サイト生成エージェント 司令塔

## 起動時の動作
このファイルを読んだら、以下の順番でファイルを読み込むこと。

1. project.md（目的・ターゲット・ゴール）
2. brand.md（世界観・デザイン・トーン）
3. skills/shiori.md（しおりページ仕様）
4. skills/warikan.md（割り勘ページ仕様）

## ステップ1：変数を順番に質問する
以下を1つずつ順番に確認すること。一度に全部聞かず、会話形式で進めること。

### 共通
1. 旅行先・都道府県（または国）
2. 旅行日程（〇月〇日〜〇泊）
3. 宿泊先（名前・URL）
4. メンバーの名前と担当係（人数分）
5. スマホ特化（max-width 420px）にするか、PC対応も含めるか
6. テーマカラーを変えるか（変える場合は希望の色・雰囲気を確認）

### しおりページ用
7. スポット候補（名前・カテゴリ・URL）
8. 1日目のタイムライン（時刻・内容）
9. 2日目のスケジュールは決まっているか
   - 決まっている場合：内容を確認する
   - 決まっていない場合：デザインマップにピンを配置する旨を伝える
10. アバター画像（外部ツールで用意後に渡してもらう）
11. デザインマップ画像（外部ツールで用意後に渡してもらう）

### 割り勘ページ用
12. Firebase接続情報（apiKey・authDomain・databaseURL・projectId）

## ステップ2：設備・持ち物リストの確認
変数が揃ったら、宿泊先URLをもとに設備・アメニティを調査し、持ち物リスト案を生成すること。
skills/shiori.md の「設備・アメニティの調査ルール」と「持ち物リストの生成ルール」に従うこと。

生成したリストを提示し、以下を確認すること。
「この内容で進めてよいですか。追加・削除があれば教えてください。」

確認が取れてからHTMLの生成に入ること。

## ステップ3：HTMLの生成

### フォルダの自動作成
HTMLを生成する前に、以下のルールでフォルダを作成しその中にファイルを格納すること。

- フォルダ名の形式：YYYYMMDD_destination
  - YYYYMMDD：旅行開始日（例：20260712）
  - destination：旅行先をローマ字・小文字・スペースなしで表記（例：hakodate、karuizawa、okinawa）
- 例：20260712_karuizawa

```bash
mkdir -p 20260712_karuizawa
```

### ファイルの生成
- skills/shiori.md に従い index.html を生成し、上記フォルダに格納すること。
- skills/warikan.md に従い warikan.html を生成し、上記フォルダに格納すること。
- brand.md のデザイン原則を両ファイルに適用すること。
- 同フォルダ内にindex.htmlが存在する場合は必ず参照してデザインを揃えること。

## ステップ4：人間による確認
生成後、以下を伝えること。

「index.html と warikan.html を生成しました。
ブラウザで開いて確認してください。
問題なければ「公開して」と伝えてください。修正があれば内容を教えてください。」

## ステップ5：公開（人間のOK後に実行）
「公開して」の指示が出たら、以下のgitコマンドを実行すること。

```bash
git add index.html warikan.html
git commit -m "旅行サイト更新"
git push origin main
```

push完了後、以下を伝えること。
「pushしました。1〜3分後に以下のURLで公開されます。
https://[GitHubユーザー名].github.io/[リポジトリ名]/」

### git未設定の場合
git remoteが設定されていない場合は、pushの前に以下を案内すること。

```bash
git init
git remote add origin https://github.com/[ユーザー名]/[リポジトリ名].git
git branch -M main
```

リモートURLはユーザーに確認すること。
認証（ユーザー名・パスワード・トークン）の入力はユーザー自身が行うこと。

## 全体を通じた注意事項
- 一度に大量の質問をしないこと。会話形式で1〜2個ずつ確認を進めること。
- 不明点はHTMLを生成する前に必ず確認を取ること。
- brand.mdのNGルールは常に守ること。
- Firebase apiKeyはソースに直書きしてよいこと（ウェブ用の公開前提のキーのため）。
