# 新しい旅行サイトを作る

## 事前準備（ユーザーに聞く前にやること）
1. `project.md`・`brand.md`・`skills/skills_shiori.md`・`skills/skills_warikan.md` を読む。
2. リポジトリ内の**最新の旅行フォルダ**（例：`20260711-12_karuizawa/`）の index.html / warikan.html を確認し、
   デザインと動的演出（ヒーローのアニメーションシーン・カウントダウン・スクロール出現・
   持ち物チェックリスト＋準備度メーター・マップのピン演出・割り勘のメンバー別収支/トースト/紙吹雪）を
   **テンプレートとして踏襲**する。ゼロから作り直さないこと。

## 質問フロー
CLAUDE.md ステップ1の変数を確認する。ただし効率化のため：

- **最初に「メンバーと係は前回と同じ？」と聞く。** 同じなら4番の質問とアバター画像の再依頼をスキップし、
  前回フォルダからアバターbase64を流用する。
- 1〜2個ずつ会話形式で聞く。選択式で聞ける質問（スマホ特化/PC対応、テーマカラー変更の有無など）は
  AskUserQuestion を使ってよい。
- 宿泊先URLをもらったらWeb調査し、設備・アメニティと持ち物リスト案を自分で作る（質問しない）。
- Firebase接続情報は「前回と同じプロジェクトを使う？」と聞き、同じなら再入力不要。

## Firebaseデータ分離（重要）
同じFirebaseプロジェクトを使い回す場合、割り勘データのパスは旅行ごとに分けること：
`db.ref('trips/<フォルダ名>/payments')`（例：`trips/20270815_okinawa/payments`）。
これで過去の旅行とデータが混ざらない。既存の軽井沢旅行（`/payments` 直下）はそのまま触らない。

## 確認チェックポイント（必ず止まる場所）
1. **持ち物リスト案の提示後**：「この内容で進めていい？追加・削除があれば教えてね」
2. **生成前の最終サマリー**：行き先・日程・メンバー・タイムライン・スポット候補を一覧で見せて
   「この内容でサイトを作るよ、いい？」
3. **生成後**：プレビューで検証した結果（スクリーンショット）を見せて確認を依頼。
   修正ラリーが終わって「公開して」と言われたら `/publish` スキルへ。

## 生成ルール
- フォルダ名：`YYYYMMDD_destination`（開始日＋ローマ字小文字）。
- `.claude/launch.json` の `--directory`（http-serverの対象フォルダ）を新フォルダに更新し、
  プレビューで表示・動作を検証してから確認依頼を出す。
- マップ・アバター画像が未提供の段階では「検討中」プレースホルダーで生成し、
  もらい次第差し替える（画像待ちで全体を止めない）。
- 大きい画像（1ファイル2MB超）は埋め込む前に縮小を提案する。
