# 公開フロー（レビュー → コミット → push）

## 1. 変更の把握
`git status --short` と `git diff --stat` で今回公開する変更を把握する。

## 2. セルフレビュー＆自動修正ループ（最大3周）
以下のチェックリストを全部通るまで「チェック→修正→再チェック」を繰り返す。
3周しても直らない問題は、無理に直さずユーザーに報告して判断を仰ぐ。

### チェックリスト
- [ ] プレースホルダーの残骸がない（`__AV1__`・`__MAP__`・`TODO`・`検討中`のまま公開してよいか確認）
- [ ] ページ間リンクが揃っている（下部ナビの index.html ↔ warikan.html、リネーム漏れがない）
- [ ] `<meta charset="UTF-8">`・viewport があり、日本語が文字化けしていない
- [ ] プレビューサーバで両ページを開き、コンソールエラーがゼロ
- [ ] 割り勘：精算ロジックの検算（合成データで収支合計が0になるか）※DBには書き込まない
- [ ] brand.md のNGルール違反がない（明朝・ゴシック系フォント、ネガティブ文言、業務感）
- [ ] 画像の重さ：1ファイル2MB超があれば縮小を提案（スマホ回線で開くため）
- [ ] Firebaseのテストモード注意コメントがコード内に残っている
- [ ] テストデータの消し忘れ：割り勘DBに「あ」「test」などの試し登録が残っていたら削除を提案

## 3. レビュー結果の報告
見つけた問題と直した内容を短くまとめて見せる。

## 4. コミットメッセージ候補の提示（必ずユーザーに選ばせる）
AskUserQuestion で候補を3つ提示する。形式の目安：
- `feat: 沖縄旅行のしおり・割り勘サイトを追加`（新規）
- `update: 軽井沢しおりのマップ画像を差し替え`（更新）
- `fix: 割り勘の精算表示を修正`（修正）
変更内容が伝わる日本語で。「その他」でユーザーが自由入力もできる。

## 5. コミット & push
```bash
git add <今回の旅行フォルダおよび変更ファイル>   # ルートの index.html ではなくフォルダ単位で
git commit -m "<選ばれたメッセージ>"
git push origin main
```
- push は必ずコミットメッセージ選択の後。勝手に push しない。
- リモート未設定ならURLをユーザーに確認してから設定する。

## 6. 完了報告
「pushしたよ！1〜3分後にここで見られるよ：
https://skkbr1113.github.io/trip-bookmaker/<フォルダ名>/」

## 7. 旅行後のリマインド
旅行終了日を過ぎた旅行の公開作業をした場合は、
「Firebase Realtime Database のルールを締める（`.read/.write: false`）かデータ削除を忘れずに」と添える。
